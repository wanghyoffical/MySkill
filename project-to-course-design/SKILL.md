# 项目课程设计

## 核心原则

把课程设计成“有状态的能力训练系统”，而不是源码导读或一次性学习路线。

先定义学习者最终必须证明自己具备什么能力，再倒推出项目模块地图。课程主轴必须来自项目本身或用户想学习的领域，而不是套固定阶段名。每个模块都必须包含概念理解、代码验证、主动练习、讲回测试、复习安排和可检查产物。

如果用户只要求课程方案，先输出方案，不要自动创建文件。只有用户明确要求“生成文件、落盘、创建课程包、写入仓库”时，才创建或修改文件。

课程包一旦落盘，必须同时生成项目根级 `START_HERE.md` 和 `AGENTS.md`。`START_HERE.md` 负责告诉学习者第一天怎么开始学；`SKILL.md` 负责指导“如何设计课程”；`AGENTS.md` 负责约束初始化后的 AI 学习会话如何读取状态、答疑、复习和推进。

## 第一步：确认最终目标

最终目的不清楚时，不要直接设计课程。

先确认：
- 最终产物：简历项目、面试能力、源码掌握、二次开发、开源贡献或内部培训。
- 学习者基础：语言、框架、领域知识和工程经验。
- 时间预算：几天、几周还是几个月。
- 深度目标：会用、会改、会复现、会解释，还是能经受面试追问。
- 学习模式：快速使用、源码理解、手写复现、实战改造、面试冲刺或综合成长。

如果答案能从项目、代码或对话上下文中确认，先阅读项目，不要直接问用户。

## 学习模式

同一个项目不能套同一套课程。根据目标选择主模式：

- 快速使用模式：优先跑通安装、配置、最小示例和常见用法。
- 源码理解模式：优先模块边界、核心路径、数据流、生命周期和扩展点。
- 手写复现模式：优先最小机制、接口抽象、关键算法和测试验证。
- 实战改造模式：优先个人改造点、迭代路线、调试记录和可展示 Demo。
- 面试冲刺模式：优先架构讲解、技术取舍、失败场景、高频追问和简历表述。
- 综合成长模式：按“使用 -> 理解 -> 复现 -> 改造 -> 表达”完整推进。

模式会影响模块深度、练习类型、最终产物和复习重点。

## 默认课程建模策略

当用户目标是综合成长、简历项目或从新人训练到能讲能改，默认采用：

```text
项目模块地图 + 模块内学习闭环 + AI 学习状态
```

具体含义：
- 项目模块地图：先按项目能力或用户目标领域拆模块，例如 Agent、Tool Use、Memory、MCP/ACP、Session、Provider、Subagent、CLI/TUI、LSP。
- 模块内学习闭环：每个模块都按“用轮子 -> 拆轮子 -> 学概念 -> 读代码 -> 理解抽象 -> 看模块关联 -> 造小轮子 -> 改轮子 -> 讲回”推进。
- AI 学习状态：用 `START_HERE.md`、`AGENTS.md`、`module-map.md`、`ai/*`、`state/*` 保持跨会话连续性和复习纪律。

如果有多个候选课程包，最终融合优先级是：
- 用事实边界更严格、状态协议更完整的一版作为底座。
- 合入模块拆分更贴近项目领域、动手训练更强的一版里的高级 lab、个人改造和 mini core 扩展。
- 保留最谨慎的 outputs 写法：只给候选模板、证据占位和达标后句式。
- 保留最具体的 mini core 黑盒合同：固定输入、期望输出、测试名、错误合同和回放合同。

不要默认生成 `stage-00-orientation`、`stage-01-run-wheel` 这类通用阶段。只有当项目本身就是线性教程，或用户明确要求按阶段推进时，才使用阶段名。通常应生成模块结构：

```text
modules/
├── 00-project-overview/
├── agent-core/
├── tool-use/
├── memory/
├── mcp-acp/
├── session/
├── provider-model/
├── subagent/
├── cli-tui/
└── lsp/
```

模块名必须来自当前项目的真实能力边界、目录结构、类型定义、文档和用户目标。不要为了凑模板强行生成项目没有的模块。

如果项目属于 coding agent、agent framework、developer tool 或 AI CLI，优先考虑这些候选模块，但仍要用当前源码确认：

```text
agent-core
tool-use
memory-or-palace
mcp-or-acp
session-persistence
provider-model
subagent-orchestration
cli-or-tui
lsp-or-editor-integration
```

最终课程顺序不是“模块越高级越靠后”的机械排序，而是：

```text
先跑通项目整体最小闭环
-> 选择第一个核心模块
-> 逐模块完成学习闭环
-> 再学习模块之间的协作关系
-> 最后做综合造轮子和改造轮子
```

## 事实边界

设计路径前必须先读项目：
- README 和文档定义项目定位与目标用户。
- 目录结构暴露模块边界。
- 类型定义、接口、核心类和测试定义真实抽象。
- 示例、脚本和测试暴露推荐使用方式和关键路径。
- issue、roadmap、博客只能补充背景，不能替代代码事实。

涉及通用技术概念时，必须补充权威资料。优先级：
- 官方文档、协议规范、标准文档。
- 官方教程、设计文档、维护者文章。
- 经典论文、书籍或被广泛引用的技术资料。
- 普通博客只能辅助理解，不能作为概念定义的唯一来源。

每个核心概念都要分三层写：
- 权威定义：这个概念在官方或标准语境中是什么意思。
- 项目实现：当前项目用哪些类型、函数、测试或配置实现它。
- 学习者解释：用简洁语言说明它为什么存在、解决什么问题、没有它会坏在哪里。

不要仅凭自然语言名词推断协议字段、结构体字段、JSON key 或架构层级。必须从代码、类型定义、配置样例或协议文档中确认。

课程中列出的源码路径必须先验证存在；通配路径也要用 `find`、`rg --files` 或等价方式确认。不存在的路径不能写进课程。

路径事实必须绑定当前源码版本。即使旧评审或旧文档说某路径不存在，也要在当前 checkout 重新验证，并在 `ai/context.md` 或 `SELF_CHECK.md` 记录当前提交或源码根。

必须分开写：
- 项目本地事实：能从当前仓库代码、测试、配置或文档确认。
- 外部框架事实：来自依赖、SDK、协议或官方文档，当前仓库不一定展开。
- 推断或待验证结论：需要运行、调试、阅读依赖源码或观察输出才能证明。

不要把某个模块的保护边界扩展到未验证的工具、进程或协议。例如文件工具有路径沙箱，不代表 shell 进程、插件、RPC 或外部框架拥有同等隔离能力，除非代码事实能证明。

若发现前面判断错误，直接说“前面判断错误，正确结论是 X”，不要用模糊措辞调和矛盾。

## 课程设计流程

### 1. 定义最终能力

使用这个格式：

```markdown
最终目标：
学完课程后，学习者能够……

最终产物：
- 可运行 Demo：
- 可讲解架构：
- 个人改造点：
- 简历描述：
- 面试问答：
- 复习与掌握记录：
```

如果目标是简历项目，必须要求学习者产出可运行项目、个人改造点、架构解释、技术取舍、失败场景和面试追问答案。

### 2. 拆解项目模块地图

优先按项目能力和用户目标领域组织课程，不要按仓库目录机械展开，也不要套通用 stage 模板。

先生成 `module-map.md`，格式如下：

```markdown
# 模块地图

## 整体最小闭环
- 项目入口：
- 最小运行命令：
- 从输入到输出的最短路径：
- 最先学习的理由：

## 模块列表
| 模块 | 为什么要学 | 权威概念 | 项目源码依据 | 前置模块 | 产物 |
|---|---|---|---|---|---|
| agent-core | 理解 agent loop 和任务执行 | Agent / Agent Loop | `path/to/file` | overview | mini agent |
| tool-use | 理解工具调用和安全边界 | Tool Use / Function Calling | `path/to/file` | agent-core | mini tool registry |
| memory | 理解记忆和上下文管理 | Memory / Retrieval | `path/to/file` | agent-core | mini memory |

## 推荐学习顺序
1. 先跑通整体最小闭环。
2. 学第一个核心模块。
3. 学支撑模块。
4. 学模块协作关系。
5. 做综合造轮子和改造轮子。
```

模块拆分规则：
- 一个模块必须对应项目中的真实能力边界，例如 Agent、Tool Use、Memory、MCP/ACP、Session、Provider、Subagent。
- 一个模块必须能落到源码证据：目录、类型、接口、函数、测试、配置或示例。
- 一个模块必须能产出练习作品：最小复现、调试记录、对比表、改造点或讲解稿。
- 模块数量宁可少而清楚，不要为了覆盖所有目录生成一堆薄模块。

### 3. 每个模块使用同一个学习闭环

每个模块目录都必须包含：

```markdown
模块名称：

学习目标：
这一节结束后，学习者能做什么。

第一性问题：
这个机制为什么存在？如果没有它，会坏在哪里？

核心概念：
只讲本模块必须理解的概念。每个概念必须包含权威定义、项目实现和学习者解释。

项目依据：
用于确认事实的文档、文件、类型、函数、测试或示例。

模块关联：
本模块依赖哪些模块，又被哪些模块调用或扩展。

代码任务：
运行、追踪、修改或复现一个最小例子，并给出可运行或可核验的验收规格。

造小轮子：
AI 必须给出完整示例代码，包括文件结构、核心代码、运行命令和期望输出。学习者可以先运行示例，再按项目源码改写。

改轮子：
学习者基于最小示例做一个个人改造点，形成简历或面试素材。

讲回测试：
学习者必须不用资料解释本模块。解释不清时，记录缺口并回到项目依据或权威资料。

复习安排：
把本模块核心概念加入 1、3、7、14、30 天复习计划。

验收证据：
命令输出、测试结果、退出码、关键输出、完整示例代码、代码 diff、调试记录、讲解稿、模块关系图或面试回答。

常见误区：
只会表层 API 调用、抽象边界错误、忽略失败场景、面试解释薄弱。
```

### 4. 设计单次学习会话

每次学习会话都按固定协议推进：

```text
读取学习状态
-> 检查到期复习
-> 定位当前模块和任务
-> 学习一个小概念或完成一个小任务
-> 做代码练习或调试验证
-> 让学习者讲回
-> 记录进度和理解缺口
-> 安排复习和下一步
```

有到期复习时，先复习，再学习新内容。不要为了推进进度跳过复习。

### 5. 使用“用轮子、拆轮子、造轮子、改轮子”模式

不要一开始就完整重写，也不要停留在框架调用。

按这个顺序推进：

```markdown
用轮子：
先跑通项目整体最小闭环，再跑通当前模块的最小用法，建立直觉。

拆轮子：
阅读当前模块的关键路径，理解模块为什么存在、输入输出是什么、失败边界在哪里。

学概念：
结合权威资料和项目源码解释核心概念，避免只背项目里的变量名。

理解抽象：
识别模块的接口、生命周期、数据流、扩展点和错误处理。

看关联：
说明当前模块和其他模块怎么协作，例如 Agent 如何调用 Tool，Tool 如何写 Session，Memory 如何影响上下文。

造小轮子：
AI 先给出完整可运行示例代码，学习者运行后再改写，复现关键机制。

改轮子：
加入个人扩展，形成可写进简历的亮点。
```

真正掌握的标准是：学习者能比较原项目模块实现和自己的最小复现版本，并讲清二者取舍、边界和模块关联。

### 6. 实战项目迭代

实战或简历模式必须有迭代路线：

```text
Build：先做最小可运行版本。
Debug：记录失败、错误、排查路径和修复依据。
Ship：整理 Demo、文档、架构说明和可展示入口。
Reflect：总结技术取舍、遗留问题和下一版改进。
```

项目日志要能直接转成简历素材和面试故事。

### 7. 造小轮子的示例代码

如果课程包含“手写复现核心机制”，不能只要求伪代码，也不能只让学习者读源码后自行实现。

AI 必须在每个需要造小轮子的模块里提供完整示例代码：
- 文件结构。
- 可复制运行的核心代码。
- 运行命令。
- 期望输出。
- 最小测试或检查命令。
- 示例代码和原项目实现的差异说明。
- 下一步让学习者改写的任务。

示例代码必须足够小，但必须完整。不要用“略”“自行实现”“参考源码补全”占位。确实无法完整实现时，要降低目标，改成更小的可运行版本。

### 8. 手写最小核心的验收

手写复现必须有验收合同。

至少给出黑盒验收合同：
- 固定输入如何触发核心流程。
- 事件、日志或状态文件必须包含哪些最小事件类型。
- 未知工具、权限失败、路径逃逸、解析失败等错误如何返回。
- replay、dry-run、resume 或等价调试能力如何验证。
- 哪些测试、命令、退出码和关键输出代表通过。

`lab.md` 和 `checks.md` 的验收口径必须一致。`lab.md` 要求 dry-run 和 replay 都通过时，`checks.md` 不能放宽成二选一；如果允许二选一，两个文件都必须写成二选一。

对于 coding agent 类项目，mini core 至少应覆盖：
- prompt event 写入 JSONL 或等价日志。
- tool call 和 tool result 写入 JSONL 或等价日志。
- 文件或命令工具的最小安全边界。
- 未知工具返回可解释错误。
- session replay、dry-run 或等价回放能力的回归测试。

## 推荐落盘结构

当用户要求生成课程文件时，优先生成这个学习包结构：

```text
course-pack/
├── START_HERE.md                      # 学习入口：今天从哪里开始
├── README.md                         # 课程入口：目标、适合人群、使用方式
├── AGENTS.md                         # 初始化后 AI 学习教练协议
├── SELF_CHECK.md                      # 生成者自检和二次评审清单
├── module-map.md                      # 项目模块地图和推荐学习顺序
├── syllabus.md                       # 总课程路线：模块、节点、时间、产物
├── tasks.md                          # 可执行任务清单
├── glossary.md                       # 核心术语表
├── modules/
│   └── module-name/
│       ├── README.md                 # 模块目标和学习顺序
│       ├── concepts.md               # 权威定义 + 项目实现 + 通俗解释
│       ├── code-reading.md           # 源码阅读路径
│       ├── relationships.md          # 与其他模块的关系
│       ├── build-wheel.md            # AI 提供的完整小轮子示例代码
│       ├── lab.md                    # 动手任务
│       ├── teach-back.md             # 讲回问题
│       └── checks.md                 # 验收证据
├── ai/
│   ├── context.md                    # 项目事实边界和关键源码路径
│   ├── coach-protocol.md             # AI 答疑和学习教练协议
│   └── ask-template.md               # 学习者提问模板
├── state/
│   ├── metadata.json                 # 学习目标、模式、当前模块、会话数
│   ├── progress.md                   # 学习日志
│   ├── review_schedule.json          # 复习队列
│   └── mastery.md                    # 掌握度清单
└── outputs/
    ├── mini-core/                     # 可选：手写复现产物入口或脚手架说明
    ├── demo.md                       # Demo 说明和运行证据
    ├── architecture-explanation.md   # 架构讲解稿
    ├── tradeoffs.md                  # 技术取舍
    ├── interview-qa.md               # 面试问答
    └── resume-bullets.md             # 简历描述
```

明确区分：
- 人读材料：`START_HERE.md`、`README.md`、`module-map.md`、`syllabus.md`、`modules/**`、`outputs/**`。
- AI 用材料：`AGENTS.md`、`ai/context.md`、`ai/coach-protocol.md`、`state/metadata.json`、`state/review_schedule.json`。
- 混合材料：`state/progress.md`、`state/mastery.md`。

`START_HERE.md` 必须回答：
- 我今天第一步做什么？
- 如果我完全不知道项目，从哪个命令开始？
- 第一个模块是什么，为什么先学它？
- 学习时要让 AI 读取哪些文件？
- 完成今天任务后要更新哪些状态文件？

## AGENTS.md 会话协议

落盘课程包时，根目录必须生成 `AGENTS.md`，用于约束初始化后的 AI 学习会话。初始化后的会话不需要反复显式声明“我引入了某个 skill”，而是必须按 `AGENTS.md` 读取课程包上下文。

`AGENTS.md` 至少包含：

```markdown
# AGENTS.md instructions

1. 回答学习者前，先读取：
   - START_HERE.md
   - module-map.md
   - state/metadata.json
   - state/review_schedule.json
   - state/progress.md
   - state/mastery.md
   - ai/context.md
   - ai/coach-protocol.md
2. 如果存在到期复习，先复习，再进入新模块或新任务。到期条件是 `next_review <= today`，不能因为 evidence 缺失就跳过；evidence 缺失时先处理为状态缺口。
3. 如果学习者不知道如何开始，先根据 `START_HERE.md` 给出今天唯一一个小任务，不要一次性倾倒全课程。
4. 学习者问“不懂、卡住、为什么、报错”时，先定位模块、文件、函数、概念或错误，再读取对应 `modules/**` 和 `ai/context.md`。
5. 先给最小提示；连续卡住时再给局部解释、最小例子或下一条证据路径。
6. 讲概念时必须结合权威资料、项目代码和通俗解释，不能只给一句定义。
7. 禁止把 README 自然语言名词当作字段、JSON key、结构体字段或协议层级。
8. 禁止把 `outputs/**` 当作学习者已掌握证据；它们默认只是候选模板。
9. 禁止跳过 teach-back；掌握度更新必须引用命令、测试、diff、调试记录或讲回证据。
10. 每次学习会话结束后更新 `state/progress.md`、`state/review_schedule.json` 和 `state/mastery.md`；如果用户只是单点咨询，可以只回答并说明未推进学习状态。
11. 如果前面回答被源码推翻，必须直接说“前面判断错误，正确结论是 X”，并给出源码依据。
12. 掌握度升级必须经过证据门槛：运行或测试证据、退出码或测试名、讲回记录、技术取舍对比。
```

## AI 实时答疑协议

AI 不是答案生成器，而是学习教练。

初始化后的答疑必须依赖 `AGENTS.md`、`state/*`、`ai/context.md` 和 `ai/coach-protocol.md`，不要要求学习者每次重新说明课程背景。

学习者说“不懂、卡住、解释一下、为什么”时，先执行：

```markdown
1. 确认位置：你卡在哪个模块、文件、函数、概念或报错？
2. 确认预期：你以为它应该怎么工作？
3. 确认实际：你实际看到什么输出、行为或代码？
4. 确认尝试：你已经查过哪些文件，试过什么？
```

回答规则：
- 先给最小提示，不直接给完整答案。
- 如果用户连续卡住，再给局部解释和最小例子。
- 解释项目事实时，引用 `ai/context.md` 或具体源码路径。
- 如果上下文缺失，直接说缺少哪个文件或事实，不要补脑。
- 区分“项目中已确认的事实”“根据代码推断的结论”“通用背景知识”。
- 涉及具体库、API、版本或外部规则时，优先查官方文档或要求用户提供材料。

## 生成后自检门槛

课程包落盘后，必须先自检再交付。

硬门槛：
- 路径存在性：课程中出现的源码路径、测试路径、脚本路径和通配路径必须可验证存在；不存在就删除、替换或标为待查证据，不能假装存在。
- 事实分层：本地源码事实、外部框架事实、运行观察和推断必须分开写。不能把依赖框架内部行为写成当前项目源码已经证明。
- 边界不泛化：某个模块的安全、状态、协议或生命周期边界只对已验证范围成立，不能扩展到 shell、子进程、插件、RPC、远程服务或其他未验证模块。
- 输出资产证据化：`outputs/demo.md`、`outputs/resume-bullets.md`、`outputs/interview-qa.md` 默认是候选模板。只有 `checks.md`、`progress.md` 或 `mastery.md` 有命令输出、测试、diff、调试记录或讲回证据后，才能改写成最终成果。
- Lab 可验收：每个 `lab.md` 必须包含命令、退出码或期望输出、测试名、最小输入输出、文件变更要求或状态文件更新规则。不能只有“阅读、画图、总结”。
- Checks 有判据：每个 `checks.md` 必须写明达标和不达标标准，不能只留下空框。
- 复习可操作：`review_schedule.json` 必须包含新增复习项、计算下次复习、记录 evidence 的样例。
- 二次评审：重要课程包必须生成 `SELF_CHECK.md`，并要求独立评审重新检查事实边界、路径存在、lab 可执行性、outputs 是否越权包装成果。
- 验收一致性：同一模块的 `lab.md`、`checks.md`、`teach-back.md` 对达标条件不能互相放宽或冲突。
- 命令成本：宽范围测试、联网下载、长耗时命令必须标注环境前提、预期耗时或失败记录方式。

`SELF_CHECK.md` 至少包含：
- 当前源码根、提交或版本标识。
- 已核验的源码路径清单和核验命令。
- 已运行或建议学习者运行的命令、退出码和关键输出。
- JSON 文件有效性检查。
- 本地事实、外部框架事实、运行观察、推断的边界摘要。
- `lab.md` 与 `checks.md` 是否一致。
- outputs 是否仍为候选模板。
- 二次评审重点：事实边界、路径存在、lab 可执行性、状态推进、outputs 越权包装。

## 复习与掌握机制

每个核心概念学完后，都要进入复习计划。

推荐 `review_schedule.json` 结构：

```json
{
  "intervals_days": [1, 3, 7, 14, 30],
  "due_selection_rule": {
    "due_if": "next_review != null && next_review <= today",
    "missing_evidence": "treat_as_state_gap_do_not_skip",
    "order": ["next_review asc", "review_count asc", "priority desc"]
  },
  "mastery_update_gate": {
    "level_3_requires": ["command_or_test_evidence", "exit_code_or_test_name", "progress_link"],
    "level_4_requires": ["teach_back_evidence", "tradeoff_comparison", "progress_link"]
  },
  "reviews": [
    {
      "concept": "核心概念名",
      "source": "modules/module-name/concepts.md",
      "module": "module-name",
      "learned_at": "YYYY-MM-DD",
      "review_count": 0,
      "next_review": "YYYY-MM-DD",
      "priority": 3,
      "status": "active",
      "evidence": "讲回记录或练习链接"
    }
  ]
}
```

推荐间隔：1、3、7、14、30 天。

完成一个模块后，必须追加或更新复习项：

```json
{
  "concept": "本模块核心概念",
  "source": "modules/module-name/concepts.md",
  "module": "module-name",
  "learned_at": "2026-05-29",
  "review_count": 0,
  "next_review": "2026-05-30",
  "priority": 3,
  "status": "active",
  "evidence": "state/progress.md#2026-05-29-module-name-teach-back"
}
```

复习通过后，先执行 `review_count += 1`，再用复习当天计算 `next_review = today + intervals_days[min(review_count, len(intervals_days)-1)]`。例如 2026-05-30 通过第一次复习后，`review_count = 1`，下一次是 2026-06-02。没有 evidence 时，不更新为掌握，也不能静默跳过到期复习项。

复习时只做主动回忆：
- 先让学习者不用资料解释。
- 解释不完整时给提示。
- 仍然不清楚时回到源码或最小例子。
- 复习后更新 `review_count`、`next_review` 和理解缺口。

## 输出资产证据追踪

`outputs/**` 初始只能是候选模板。每一条 Demo、架构结论、取舍、面试答案或简历句式都要保留证据槽：

```markdown
证据编号：
来源：
命令/测试：
退出码或关键输出：
源码路径：
讲回记录：
```

没有证据编号的内容只能保留在候选区，不能改写成最终成果。简历句式优先写成“达标后可写：……[证据占位]”，避免学习者提前复制成已完成项目。

## 建议后置内容的判断

满足任一条件就后置：
- 不服务当前学习模式。
- 不影响最小运行闭环。
- 不影响最终产物。
- 当前模块无法通过代码验证。
- 属于优化、边缘场景、历史背景或高级扩展。

后置不是删除，必须说明后置原因和重新学习的触发条件。

## 常见错误

- 按仓库目录讲文件，而不是按能力成长设计课程。
- 套固定 stage 模板，而不是先按项目或用户目标拆模块。
- 只生成路线图，没有状态、复习、日志和掌握证据。
- 生成了很多文件，但没有 `START_HERE.md` 告诉用户第一天怎么开始。
- 还没跑通最小闭环，就堆太多概念。
- 概念讲解只给几句浅定义，没有查官方资料，也没有结合项目代码。
- 只教框架 API，不解释 API 背后的机制。
- 造小轮子时只让用户读源码自己写，没有提供完整可运行示例代码。
- 学习者没讲回，就标记“掌握”。
- 只做玩具 Demo，不进入真实项目。
- 缺少个人改造点，导致简历项目没有辨识度。
- 结尾没有面试问题、技术取舍和失败场景。
- 落盘课程包后没有 `AGENTS.md`，导致后续 AI 不读取状态和协议。
- 用不存在的源码路径、测试路径或通配路径设计学习任务。
- 把外部框架推断写成本项目源码事实。
- 把候选简历、Demo 或面试答案写成学习者已经完成的成果。
- `lab.md` 只有阅读或总结任务，没有命令、测试、输入输出或状态更新判据。
- 没有二次事实评审，导致事实错误进入 `ai/context.md` 后被后续会话反复放大。
- 只生成 `SELF_CHECK.md`，但没有列出独立二次评审 checklist。
- `review_schedule.json` 有日期样例但和计算规则不一致。
- 到期复习项缺 evidence 时被静默跳过，而不是作为状态缺口处理。
- `lab.md` 要求多个验收证据，`checks.md` 却放宽为任意一个。
- outputs 的简历句式太完整，虽然标注候选但仍容易被学习者提前复制。
