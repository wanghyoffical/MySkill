# 项目课程设计

## 核心原则

把课程设计成“有状态的能力训练系统”，而不是源码导读或一次性学习路线。

先定义学习者最终必须证明自己具备什么能力，再倒推出阶段节点。每个节点都必须包含概念理解、代码验证、主动练习、讲回测试、复习安排和可检查产物。

如果用户只要求课程方案，先输出方案，不要自动创建文件。只有用户明确要求“生成文件、落盘、创建课程包、写入仓库”时，才创建或修改文件。

课程包一旦落盘，必须同时生成项目根级 `AGENTS.md`。`SKILL.md` 负责指导“如何设计课程”，`AGENTS.md` 负责约束初始化后的 AI 学习会话如何读取状态、答疑、复习和推进。

## 第一步：确认最终目标

最终目的不清楚时，不要直接设计课程。

先确认：
- 最终产物：简历项目、面试能力、源码掌握、二次开发、开源贡献或内部培训。
- 学习者基础：语言、框架、领域知识和工程经验。
- 时间预算：几天、几周还是几个月。
- 深度目标：会用、会改、会复现、会解释，还是能经受面试追问。
- 学习模式：快速使用、源码理解、手写复现、实战改造、面试冲刺或综合成长。

如果答案能从项目、代码或对话上下文中确认，先阅读项目，不要直接问用户。

## 学习模式

同一个项目不能套同一套课程。根据目标选择主模式：

- 快速使用模式：优先跑通安装、配置、最小示例和常见用法。
- 源码理解模式：优先模块边界、核心路径、数据流、生命周期和扩展点。
- 手写复现模式：优先最小机制、接口抽象、关键算法和测试验证。
- 实战改造模式：优先个人改造点、迭代路线、调试记录和可展示 Demo。
- 面试冲刺模式：优先架构讲解、技术取舍、失败场景、高频追问和简历表述。
- 综合成长模式：按“使用 -> 理解 -> 复现 -> 改造 -> 表达”完整推进。

模式会影响阶段深度、练习类型、最终产物和复习重点。

## 默认融合策略

当用户目标是综合成长、简历项目或从新人训练到能讲能改，默认采用：

```text
动手主干 + AI 学习状态 + 高级架构补全
```

具体含义：
- 动手主干：先让学习者能运行、追踪、复现、改造和讲回，不先做源码百科。
- AI 学习状态：用 `AGENTS.md`、`ai/*`、`state/*` 保持跨会话连续性和复习纪律。
- 高级架构补全：只补服务最终能力证明的高级模块，不展开所有边缘实现。

默认阶段结构：

```text
stage-00-orientation
stage-01-run-wheel
stage-02-trace-core-flow
stage-03-mini-core
stage-04-personal-mod
stage-05-advanced-orchestration
stage-06-state-memory-or-persistence
stage-07-interfaces-ui-or-protocols
stage-08-architecture-expression
```

`stage-00` 到 `stage-04` 必须形成“用轮子、拆轮子、造小轮子、改轮子”的最短闭环。`stage-05` 到 `stage-07` 只纳入对项目价值、工程能力或面试表达有直接贡献的高级模块。`stage-08` 统一收束 Demo、架构讲解、取舍、面试问答和简历素材。

如果项目属于 coding agent、agent framework、developer tool 或 AI CLI，优先使用这个更具体的模板：

```text
stage-00-orientation
stage-01-run-wheel
stage-02-trace-tools-session
stage-03-mini-agent-core
stage-04-personal-mod
stage-05-subagent-orchestration
stage-06-memory-palace
stage-07-lsp-tui-rpc-acp
stage-08-architecture-expression
```

其中 `stage-02` 必须追踪最小运行链路和状态落盘边界；`stage-03` 必须手写可测试的 mini core；`stage-05` 到 `stage-07` 只补能证明工程深度的高级模块，例如 subagent 编排、记忆/持久化、LSP/TUI/RPC/ACP 边界。

## 事实边界

设计路径前必须先读项目：
- README 和文档定义项目定位与目标用户。
- 目录结构暴露模块边界。
- 类型定义、接口、核心类和测试定义真实抽象。
- 示例、脚本和测试暴露推荐使用方式和关键路径。
- issue、roadmap、博客只能补充背景，不能替代代码事实。

不要仅凭自然语言名词推断协议字段、结构体字段、JSON key 或架构层级。必须从代码、类型定义、配置样例或协议文档中确认。

课程中列出的源码路径必须先验证存在；通配路径也要用 `find`、`rg --files` 或等价方式确认。不存在的路径不能写进课程。

必须分开写：
- 项目本地事实：能从当前仓库代码、测试、配置或文档确认。
- 外部框架事实：来自依赖、SDK、协议或官方文档，当前仓库不一定展开。
- 推断或待验证结论：需要运行、调试、阅读依赖源码或观察输出才能证明。

不要把某个模块的保护边界扩展到未验证的工具、进程或协议。例如文件工具有路径沙箱，不代表 shell 进程、插件、RPC 或外部框架拥有同等隔离能力，除非代码事实能证明。

若发现前面判断错误，直接说“前面判断错误，正确结论是 X”，不要用模糊措辞调和矛盾。

## 课程设计流程

### 1. 定义最终能力

使用这个格式：

```markdown
最终目标：
学完课程后，学习者能够……

最终产物：
- 可运行 Demo：
- 可讲解架构：
- 个人改造点：
- 简历描述：
- 面试问答：
- 复习与掌握记录：
```

如果目标是简历项目，必须要求学习者产出可运行项目、个人改造点、架构解释、技术取舍、失败场景和面试追问答案。

### 2. 拆解能力地图

优先按能力成长顺序组织课程，不要按仓库目录机械展开。

默认阶段阶梯：

```markdown
阶段 0：项目定位
理解项目解决什么问题，以及为什么值得学习。

阶段 1：最小运行闭环
跑通最小输入到输出路径。

阶段 2：核心概念
只学习解释最小闭环必须用到的概念。

阶段 3：代码验证
追踪、调试或修改实现这些概念的关键代码。

阶段 4：框架理解
识别模块、接口、扩展点、生命周期和数据流。

阶段 5：复现核心机制
手写关键机制的最小版本。

阶段 6：真实项目
构建或扩展一个完整、可展示的应用。

阶段 7：表达与面试
沉淀简历表述、架构讲解、技术取舍和高频追问答案。
```

### 3. 每个节点使用同一个学习闭环

每个阶段节点都必须包含：

```markdown
节点名称：

学习目标：
这一节结束后，学习者能做什么。

第一性问题：
这个机制为什么存在？如果没有它，会坏在哪里？

核心概念：
只讲本节点必须理解的概念。

项目依据：
用于确认事实的文档、文件、类型、函数、测试或示例。

代码任务：
运行、追踪、修改或复现一个最小例子，并给出可运行或可核验的验收规格。

讲回测试：
学习者必须不用资料解释本节点。解释不清时，记录缺口并回到项目依据。

复习安排：
把本节点核心概念加入 1、3、7、14、30 天复习计划。

验收证据：
命令输出、测试结果、退出码、关键输出、代码 diff、调试记录、讲解稿、架构图或面试回答。

常见误区：
只会表层 API 调用、抽象边界错误、忽略失败场景、面试解释薄弱。
```

### 4. 设计单次学习会话

每次学习会话都按固定协议推进：

```text
读取学习状态
-> 检查到期复习
-> 定位当前阶段和节点
-> 学习一个小概念或完成一个小任务
-> 做代码练习或调试验证
-> 让学习者讲回
-> 记录进度和理解缺口
-> 安排复习和下一步
```

有到期复习时，先复习，再学习新内容。不要为了推进进度跳过复习。

### 5. 使用“用轮子、拆轮子、造轮子、改轮子”模式

不要一开始就完整重写，也不要停留在框架调用。

按这个顺序推进：

```markdown
用轮子：
先跑通现有项目或框架，建立整体直觉。

拆轮子：
阅读关键路径，理解模块为什么存在。

造小轮子：
手写最小版本，复现关键机制。

改轮子：
加入个人扩展，形成可写进简历的亮点。
```

真正掌握的标准是：学习者能比较原项目实现和自己的最小复现版本，并讲清二者取舍。

### 6. 实战项目迭代

实战或简历模式必须有迭代路线：

```text
Build：先做最小可运行版本。
Debug：记录失败、错误、排查路径和修复依据。
Ship：整理 Demo、文档、架构说明和可展示入口。
Reflect：总结技术取舍、遗留问题和下一版改进。
```

项目日志要能直接转成简历素材和面试故事。

### 7. 手写最小核心的验收

如果课程包含“手写复现核心机制”，不能只要求伪代码。

至少给出黑盒验收合同：
- 固定输入如何触发核心流程。
- 事件、日志或状态文件必须包含哪些最小事件类型。
- 未知工具、权限失败、路径逃逸、解析失败等错误如何返回。
- replay、dry-run、resume 或等价调试能力如何验证。
- 哪些测试、命令、退出码和关键输出代表通过。

对于 coding agent 类项目，mini core 至少应覆盖：
- prompt event 写入 JSONL 或等价日志。
- tool call 和 tool result 写入 JSONL 或等价日志。
- 文件或命令工具的最小安全边界。
- 未知工具返回可解释错误。
- session replay、dry-run 或等价回放能力的回归测试。

## 推荐落盘结构

当用户要求生成课程文件时，优先生成这个学习包结构：

```text
course-pack/
├── README.md                         # 课程入口：目标、适合人群、使用方式
├── AGENTS.md                         # 初始化后 AI 学习教练协议
├── syllabus.md                       # 总课程路线：阶段、节点、时间、产物
├── tasks.md                          # 可执行任务清单
├── glossary.md                       # 核心术语表
├── stages/
│   └── stage-xx-name/
│       ├── README.md                 # 阶段目标
│       ├── concepts.md               # 核心概念
│       ├── code-reading.md           # 源码阅读路径
│       ├── lab.md                    # 动手任务
│       ├── teach-back.md             # 讲回问题
│       └── checks.md                 # 验收证据
├── ai/
│   ├── context.md                    # 项目事实边界和关键源码路径
│   ├── coach-protocol.md             # AI 答疑和学习教练协议
│   └── ask-template.md               # 学习者提问模板
├── state/
│   ├── metadata.json                 # 学习目标、模式、当前阶段、会话数
│   ├── progress.md                   # 学习日志
│   ├── review_schedule.json          # 复习队列
│   └── mastery.md                    # 掌握度清单
└── outputs/
    ├── demo.md                       # Demo 说明和运行证据
    ├── architecture-explanation.md   # 架构讲解稿
    ├── tradeoffs.md                  # 技术取舍
    ├── interview-qa.md               # 面试问答
    └── resume-bullets.md             # 简历描述
```

明确区分：
- 人读材料：`README.md`、`syllabus.md`、`stages/**`、`outputs/**`。
- AI 用材料：`AGENTS.md`、`ai/context.md`、`ai/coach-protocol.md`、`state/metadata.json`、`state/review_schedule.json`。
- 混合材料：`state/progress.md`、`state/mastery.md`。

## AGENTS.md 会话协议

落盘课程包时，根目录必须生成 `AGENTS.md`，用于约束初始化后的 AI 学习会话。初始化后的会话不需要反复显式声明“我引入了某个 skill”，而是必须按 `AGENTS.md` 读取课程包上下文。

`AGENTS.md` 至少包含：

```markdown
# AGENTS.md instructions

1. 回答学习者前，先读取：
   - state/metadata.json
   - state/review_schedule.json
   - state/progress.md
   - state/mastery.md
   - ai/context.md
   - ai/coach-protocol.md
2. 如果存在到期复习，先复习，再进入新阶段。
3. 学习者问“不懂、卡住、为什么、报错”时，先定位阶段、文件、函数、概念或错误，再读取对应 `stages/**` 和 `ai/context.md`。
4. 先给最小提示；连续卡住时再给局部解释、最小例子或下一条证据路径。
5. 禁止把 README 自然语言名词当作字段、JSON key、结构体字段或协议层级。
6. 禁止把 `outputs/**` 当作学习者已掌握证据；它们默认只是候选模板。
7. 禁止跳过 teach-back；掌握度更新必须引用命令、测试、diff、调试记录或讲回证据。
8. 每次学习会话结束后更新 `state/progress.md`、`state/review_schedule.json` 和 `state/mastery.md`；如果用户只是单点咨询，可以只回答并说明未推进学习状态。
9. 如果前面回答被源码推翻，必须直接说“前面判断错误，正确结论是 X”，并给出源码依据。
```

## AI 实时答疑协议

AI 不是答案生成器，而是学习教练。

初始化后的答疑必须依赖 `AGENTS.md`、`state/*`、`ai/context.md` 和 `ai/coach-protocol.md`，不要要求学习者每次重新说明课程背景。

学习者说“不懂、卡住、解释一下、为什么”时，先执行：

```markdown
1. 确认位置：你卡在哪个阶段、文件、函数、概念或报错？
2. 确认预期：你以为它应该怎么工作？
3. 确认实际：你实际看到什么输出、行为或代码？
4. 确认尝试：你已经查过哪些文件，试过什么？
```

回答规则：
- 先给最小提示，不直接给完整答案。
- 如果用户连续卡住，再给局部解释和最小例子。
- 解释项目事实时，引用 `ai/context.md` 或具体源码路径。
- 如果上下文缺失，直接说缺少哪个文件或事实，不要补脑。
- 区分“项目中已确认的事实”“根据代码推断的结论”“通用背景知识”。
- 涉及具体库、API、版本或外部规则时，优先查官方文档或要求用户提供材料。

## 生成后自检门槛

课程包落盘后，必须先自检再交付。

硬门槛：
- 路径存在性：课程中出现的源码路径、测试路径、脚本路径和通配路径必须可验证存在；不存在就删除、替换或标为待查证据，不能假装存在。
- 事实分层：本地源码事实、外部框架事实、运行观察和推断必须分开写。不能把依赖框架内部行为写成当前项目源码已经证明。
- 边界不泛化：某个模块的安全、状态、协议或生命周期边界只对已验证范围成立，不能扩展到 shell、子进程、插件、RPC、远程服务或其他未验证模块。
- 输出资产证据化：`outputs/demo.md`、`outputs/resume-bullets.md`、`outputs/interview-qa.md` 默认是候选模板。只有 `checks.md`、`progress.md` 或 `mastery.md` 有命令输出、测试、diff、调试记录或讲回证据后，才能改写成最终成果。
- Lab 可验收：每个 `lab.md` 必须包含命令、退出码或期望输出、测试名、最小输入输出、文件变更要求或状态文件更新规则。不能只有“阅读、画图、总结”。
- Checks 有判据：每个 `checks.md` 必须写明达标和不达标标准，不能只留下空框。
- 复习可操作：`review_schedule.json` 必须包含新增复习项、计算下次复习、记录 evidence 的样例。
- 二次评审：重要课程包应由独立评审重新检查事实边界、路径存在、lab 可执行性、outputs 是否越权包装成果。

## 复习与掌握机制

每个核心概念学完后，都要进入复习计划。

推荐 `review_schedule.json` 结构：

```json
{
  "intervals_days": [1, 3, 7, 14, 30],
  "reviews": [
    {
      "concept": "核心概念名",
      "source": "stages/stage-xx/concepts.md",
      "learned_at": "YYYY-MM-DD",
      "review_count": 0,
      "next_review": "YYYY-MM-DD",
      "evidence": "讲回记录或练习链接"
    }
  ]
}
```

推荐间隔：1、3、7、14、30 天。

完成一个阶段后，必须追加或更新复习项：

```json
{
  "concept": "本阶段核心概念",
  "source": "stages/stage-xx/concepts.md",
  "learned_at": "2026-05-29",
  "review_count": 0,
  "next_review": "2026-05-30",
  "evidence": "state/progress.md#2026-05-29-stage-xx-teach-back"
}
```

复习通过后，`review_count += 1`，`next_review` 按 `intervals_days[min(review_count, len(intervals_days)-1)]` 推进。没有 evidence 时，不更新为掌握。

复习时只做主动回忆：
- 先让学习者不用资料解释。
- 解释不完整时给提示。
- 仍然不清楚时回到源码或最小例子。
- 复习后更新 `review_count`、`next_review` 和理解缺口。

当用户只要求课程设计时，输出：

```markdown

## 3. 学习者基础与前置要求
## 4. 项目核心价值
## 5. 能力地图
```

## 建议后置内容的判断

满足任一条件就后置：
- 不服务当前学习模式。
- 不影响最小运行闭环。
- 不影响最终产物。
- 当前阶段无法通过代码验证。
- 属于优化、边缘场景、历史背景或高级扩展。

后置不是删除，必须说明后置原因和重新学习的触发条件。

## 常见错误

- 按仓库目录讲文件，而不是按能力成长设计课程。
- 只生成路线图，没有状态、复习、日志和掌握证据。
- 还没跑通最小闭环，就堆太多概念。
- 只教框架 API，不解释 API 背后的机制。
- 学习者没讲回，就标记“掌握”。
- 只做玩具 Demo，不进入真实项目。
- 缺少个人改造点，导致简历项目没有辨识度。
- 结尾没有面试问题、技术取舍和失败场景。
- 落盘课程包后没有 `AGENTS.md`，导致后续 AI 不读取状态和协议。
- 用不存在的源码路径、测试路径或通配路径设计学习任务。
- 把外部框架推断写成本项目源码事实。
- 把候选简历、Demo 或面试答案写成学习者已经完成的成果。
- `lab.md` 只有阅读或总结任务，没有命令、测试、输入输出或状态更新判据。
- 没有二次事实评审，导致事实错误进入 `ai/context.md` 后被后续会话反复放大。
